# `/carbon-streamlit` Skill Feedback

Feedback from testing the skill during the City Explorer project implementation.

---

## Overall Assessment: Good Foundation

The skill successfully generated a working frontend with IBM Carbon Design styling. It provided a solid baseline that required targeted modifications rather than a rewrite.

### What Worked Well

1. **Clean separation of concerns** - Each file has a single responsibility:
   - `styles.py` - CSS generation only
   - `components.py` - HTML rendering only
   - `api_client.py` - Network logic only
   - `config.py` - Configuration only
   - `app.py` - App orchestration only

2. **Parameterized theming** - The `THEME` dict in `styles.py` is well-designed:
   - All IBM Carbon colors in one place
   - `get_carbon_css(theme)` accepts overrides
   - Easy to customize without touching CSS strings

3. **Efficient token usage** - Templates are lean (~355 lines total):
   - No bloated dependencies
   - Minimal CSS (only what's needed)
   - Clean Python with no unnecessary abstractions

4. **Good utility functions**:
   - `dedup_tools()` prevents duplicate cards
   - `_normalize_args()` handles various input formats robustly
   - `escape()` used consistently for XSS protection

5. **SSE client implementation**:
   - Generator pattern is memory-efficient
   - Proper error handling (ConnectError, ReadTimeout, HTTPStatusError)
   - `StreamResult` dataclass cleanly accumulates metadata

## Bugs to Fix

### 1. Cached Health Check Placement (High Priority)

**Location:** `app.py` lines 67-69

**Problem:** The `@st.cache_data` decorator on `_cached_health` is defined inside the sidebar `with` block. This causes the function to be redefined on every Streamlit rerun, which defeats caching and may cause issues.

**Fix:** Move the cached function to module level (outside any blocks):

```python
# At module level, after imports
@st.cache_data(ttl=30)
def _cached_health(base_url: str) -> dict | None:
    return check_health(base_url)

# Then in sidebar:
with st.sidebar:
    # ...
    health = _cached_health(config.api_base_url)
```

### 2. Missing `__init__.py` Template

**Problem:** The skill instructions mention generating `__init__.py` but no template file exists in the templates directory.

**Fix:** Add `templates/__init__.py` with a simple comment:
```python
# Frontend package
```

## Architecture Recommendations

### 1. Separate Design System from Layout (High Priority)

**Current behavior:** The skill generates a specific layout (centered, sidebar, Logo → Title → Form → Response structure).

**Problem:** Users may want Carbon styling for different layouts (full-width chat, dashboard, multi-column, etc.).

**Recommendation:** Reframe the skill as a "design system" not a "layout template":
- `styles.py` - IBM Carbon design system (colors, typography, component styles)
- `components.py` - Reusable styled components
- `api_client.py` - Optional SSE client (useful but separate concern)
- `app.py` - Provide as an **example** only, not generated by default

The skill should ask: "Would you like me to generate an example app.py?" rather than always including it.

### 2. Make `app.py` Optional or Modular

Instead of one monolithic app template, consider:
- `app_chat.py` - Chat-focused layout (like the current one)
- `app_form.py` - Form-submission focused layout
- Let users choose or skip entirely

### 3. Add `__init__.py` with Convenience Imports

The generated `__init__.py` should export key items for cleaner imports:
```python
from .styles import get_carbon_css, THEME
from .components import render_tool_card, render_health_indicator, render_empty_state
from .config import config, FrontendConfig
from .api_client import stream_chat, check_health, StreamResult
```

## Feature Requests

### 1. Dark Mode Support (Medium Priority)

IBM Carbon has both light and dark themes. Add:
```python
THEME_DARK: dict[str, str] = {
    "primary": "#78a9ff",
    "bg_primary": "#161616",
    "bg_secondary": "#262626",
    "text_primary": "#f4f4f4",
    # ... etc
}
```

And a parameter to `get_carbon_css(theme=None, dark=False)`.

### 2. Modular CSS Generation (Medium Priority)

Split `get_carbon_css()` into composable parts:
```python
def get_base_css(theme) -> str:
    """Fonts, colors, global resets"""

def get_component_css(theme) -> str:
    """Tool cards, health indicators, skill cards"""

def get_layout_css(theme) -> str:
    """Optional layout-specific styles"""

def get_carbon_css(theme=None, include_layout=True) -> str:
    """Combine all CSS"""
```

This lets users pick and choose what they need.

### 3. Additional Components (Low Priority)

Expand `components.py` with more Carbon-styled components:
- `render_message_bubble()` - For chat interfaces
- `render_status_badge()` - For status indicators
- `render_notification()` - For toast/banner messages
- `render_card()` - Generic content card

### 4. Self-Hosted Fonts Option (Low Priority)

The current implementation loads IBM Plex fonts from Google Fonts CDN:
```css
@import url('https://fonts.googleapis.com/css2?family=IBM+Plex+Sans...');
```

This adds external dependency and slight latency. Offer an option to:
- Use system fonts (`-apple-system, BlinkMacSystemFont, 'Segoe UI', ...`)
- Bundle font files locally
- Keep CDN (default)

## Skill Instruction Updates

### 1. Don't Force Layout

Current behavior asks for "sidebar title" implying a sidebar is required.

Better approach:
1. Ask what type of app (chat, form, dashboard, custom)
2. Generate only styling files for "custom"
3. Generate example app.py for specific types

### 2. Ask About Optional Features

Add questions:
- "Include SSE streaming client?" (Yes/No - some apps don't need it)
- "Include example app.py?" (Yes/No)
- "Theme preference?" (Light/Dark/Both)

### 3. Provide More Customization Hooks

After generation, remind users:
- "Customize colors in the THEME dict in styles.py"
- "Add new components to components.py following the existing pattern"
- "The api_client.py is optional - delete if using a different backend pattern"

---

## Summary

The `/carbon-streamlit` skill is **usable and valuable**. The main improvements are:
1. **Bug fix**: Move cached health check to module level
2. **Architecture**: Separate design system from layout templates
3. **Features**: Add dark mode, modular CSS, more components
4. **Docs**: Clarify it's a design system, not a layout generator

These changes would make the skill more flexible while keeping it simple and efficient.
